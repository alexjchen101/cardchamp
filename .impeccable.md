## Design Context

### Product

**Photon Collective · CardChamp**
An internal admin tool for the CardChamp ops team. Surfaces merchant sync state, volume data, owner assignments, and manual sync controls — all backed by the CoPilot → HubSpot pipeline. Operators use it to monitor, troubleshoot, and act, not to be impressed.

---

### Users

- **Who**: Internal CardChamp ops/admin staff
- **Context**: Desktop, business hours. They open it to check whether a sync ran correctly, fix a misassigned owner, look up a specific merchant, or trigger a manual re-sync.
- **Disposition**: Technical-enough to understand sync errors; not engineers. They want direct answers: did it work, what went wrong, which merchant, fix it.
- **Decision pattern**: Check sync log → find the merchant → see what failed → trigger fix or update owner map

---

### Jobs to be done

1. **Sync status log** — see every sync run: who synced, when, errors, what changed
2. **Merchant lookup** — search by name/ID, see current CoPilot data, HubSpot sync state, deal stage
3. **Owner map** — manage sales code → HubSpot owner assignments; edit inline
4. **Manual trigger** — run a sync for one merchant or a full batch from the UI
5. **Volume data** — browse aggregated daily SFTP drops (MTD/YTD volumes per merchant)

---

### Suite Membership

CardChamp is part of the **Photon Collective** product suite alongside Regulatory Intelligence, Ballantyne, and Krystal. The full suite design system applies:

#### App bar (non-negotiable)
- Full-width, 44px fixed height, `--accent` background (deep forest green `#1A3A2A`)
- Left: `Photon Collective` in DM Serif Display + `CardChamp` in DM Sans, subdued, smaller
- Right: workflow navigation labels (Sync Log · Merchants · Owners · Volume)
- Never absent. Never icon-only nav.

#### Color tokens (source of truth)
```css
--bg:        #F7F5F0;   /* Warm parchment surface */
--surface:   #FFFFFF;   /* Cards, panels */
--text:      #0F0F0E;
--text-2:    #5C5A55;
--text-3:    #9B9890;
--accent:    #1A3A2A;   /* Deep forest green */
--accent-lt: #E8F0EB;
--cite:      #2D5A3D;
--border:    #E2DFD8;
--border-2:  #D0CCC4;
--warn:      #8B2E00;   /* Burnt orange — errors, failures */
--warn-lt:   #FDF0E8;
```

#### Typography
- `DM Serif Display` — app bar wordmark only
- `DM Sans` — all UI, body, navigation, labels
- `JetBrains Mono` — merchant IDs, CoPilot IDs, sales codes, error codes, timestamps, `code` literals
- Font size floor: 13px. Never sacrifice legibility.

#### Card style
- `background: --surface`, `border: 1px solid --border`, `border-radius: 6px`, `box-shadow: --shadow-sm`
- No coloured card backgrounds (except `--warn-lt` for error states)
- No heavy drop shadows

### Brand Personality

Three words (inherited from suite): **Sharp. Authoritative. Restrained.**

This is an ops instrument, not a dashboard product. It doesn't need to look impressive — it needs to be fast and correct. Errors are surfaced clearly. Success is quiet. No celebrating when a sync completes.

---

### Aesthetic Direction

**Theme**: Light only. Office hours, desktop.

**The feeling to hit**: A well-made internal database admin — closer to Retool's density and directness than to a BI dashboard with vanity charts. Like a senior engineer designed it for their own team: zero decoration, total clarity.

**Anti-references** — must not look like:
- A generic SaaS admin panel: grey everywhere, big metric cards, donut charts
- A Streamlit app: widget-heavy, no visual hierarchy
- A Grafana dashboard: too much chart chrome, dark defaults
- A consumer CRM: friendly copy, large avatars, pastel accents

---

### Layout Principles (CardChamp-specific)

The app bar is the only invariant. Below it, each workflow gets first-principles layout:

**Sync Log** — table is the hero. Full-width, dense, sortable. Status chips (`LIVE`, `ERROR`, `SKIPPED`) use `JetBrains Mono` + `--accent-lt` / `--warn-lt` backgrounds. Row detail expands inline or opens a right panel.

**Merchant Lookup** — search bar prominent at top. Results as a scannable table. Selecting a merchant opens a detail view: CoPilot data on the left, HubSpot sync state on the right. Manual trigger button in the detail view, never buried.

**Owner Map** — table with inline editing. Sales code column in `JetBrains Mono`. Clicking an owner cell opens an inline select. Save is per-row, not a global form submit. Add/remove rows at the bottom.

**Volume** — daily SFTP data aggregated per merchant. Table-first (not charts). Optional sparkline column acceptable if it genuinely aids comparison — not decorative.

---

### Workflow-Specific Design Principles

1. **The table is the surface** — every workflow starts with a scannable table. Do not bury data in cards.
2. **Inline editing is the primary interaction** — owner map cells, trigger buttons — operate without leaving context.
3. **Errors are first-class** — `--warn` / `--warn-lt` must be immediately visible. Don't bury failures in expandable rows.
4. **IDs are not prose** — every CoPilot ID, HubSpot ID, sales code, and MID renders in `JetBrains Mono`. They are identifiers, not text.
5. **Manual trigger is one click** — the most common action (re-sync a merchant) must never require a modal or multi-step flow.

---

### Design Invariants (CardChamp-specific)

1. **App bar**: `Photon Collective` · `CardChamp` — same format as suite. Non-negotiable.
2. **No dark mode** — light only, consistent with suite.
3. **Tables over cards** — ops data belongs in tables, not card grids.
4. **Filters always visible** — search/filter controls do not collapse. Always reachable without a click.
5. **Inline editing without noise** — hover state signals editability; cells are clean at rest.
6. **Error states use `--warn` palette** — never generic red. Burnt orange is the suite's error register.
7. **No sidebar required** — workflow nav lives in the app bar. Below the bar: full-bleed table surface with an optional filter strip above.

---

### Design Principles

1. **Ops first** — every decision serves speed-to-answer: what synced, what failed, who owns it, fix it.
2. **Identity in monospace** — merchant IDs, sales codes, system codes always in `JetBrains Mono`. They're identifiers, not decoration.
3. **Quiet success, loud failure** — successful syncs are a row in the log. Errors get colour, prominence, and a clear action path.
4. **Instrument, not wizard** — no step-by-step onboarding flows. The operator knows what they're doing. Get out of the way.
5. **Suite coherence** — every screen should look like it belongs next to Regulatory Intelligence and Ballantyne. Shared tokens, shared app bar, shared type system.
